**システム名**: ASOPokeTool（ポケモンカード大会運営ツール）

## 1. システム概要

### 1.1 目的

ポケモンカードゲームの大会を効率的に運営するための Web アプリケーション。  
スイスドロー方式による対戦マッチング、チェックイン管理、決勝トーナメント運営、ポイント集計など、大会運営に必要な一連の機能を提供する。

### 1.2 対象ユーザー

| ユーザー種別 | 説明 |
|---|---|
| 参加者（ゲスト） | アカウント未登録でも大会情報閲覧・名前入力でエントリー可能 |
| 参加者（会員） | アカウント登録済み。ワンクリックエントリー・過去大会履歴参照が可能 |
| 運営者（管理者） | 大会を設定・管理する運営スタッフ |

### 1.3 主要機能一覧

| No. | 機能名 | 概要 |
|---|---|---|
| 1 | 大会エントリー機能 | 参加者が大会に登録・辞退する |
| 2 | チェックイン機能 | QRコードによる当日チェックイン |
| 3 | スイスドローマッチング | 同勝利数でのマッチング自動生成 |
| 4 | 対戦結果登録 | 勝者が結果を登録する |
| 5 | マッチング表示 | 参加者・管理者向けの対戦表示 |
| 6 | 大会棄権機能 | 途中ドロップの管理 |
| 7 | ランキング表示 | スイスドロー順位の表示 |
| 8 | 決勝トーナメント | スイスドロー結果に基づくトーナメント運営 |
| 9 | 累計ポイント | 大会ごとのポイント計算・集計 |
| 10 | 累計ポイントランキング | 全大会を通じたポイントランキング |
| 11 | 大会作成・設定 | 大会設定（ポイント・入賞等）の管理 |
| 12 | 対戦タイマー・BGM | 対戦時間のカウントダウン管理（BGMおよび背景画像アップロード） |
| 13 | アカウント機能 | 参加者向けアカウント登録・自動ログイン・過去大会履歴・ワンクリックエントリー |
| 14 | プロジェクター投影対戦表 | スクリーン投影用にナビを排除し、卓番号と対戦者を巨大表示する画面（自動更新付） |
| 15 | 結果画像保存機能 | Teams共有用に、予選順位表や決勝トーナメントツリーをワンクリックでPNG画像ダウンロード |
| 16 | 選手手動一括登録 | カンマや改行区切りのCSV形式テキストによる選手の一括登録 |
| 17 | 管理者・ユーザー管理 | 管理者アカウント追加、全ユーザーのサマリー一覧およびパスワードの強制上書きリセット |

## 5. 主要ビジネスロジック

### 5.1 スイスドローアルゴリズム

#### 5.1.1 基本ルール

| ルール | 内容 |
|---|---|
| **同点優先** | 勝ち点・勝利数が同じプレイヤー同士を優先的にマッチングする |
| **再戦回避** | 同じ大会内で一度対戦したプレイヤー同士は原則として再マッチングしない |
| **不戦勝（BYE）のケア** | 参加人数が奇数で余りが出た場合に BYE を発生させる。BYE は大会を通じて 1 人につき 1 回までとなるよう調整する |

#### 5.1.2 アルゴリズムの流れ

```
STEP 1: 成績の集計とソート
  - 各ラウンド終了後に勝ち点を集計する
  - 勝ち点（降順）→ OMW%（降順）の順でプレイヤーをソートする

STEP 2: スコアグループの作成
  - 同じ勝ち点のプレイヤーをグループ化する
    例: 【3勝0敗グループ】【2勝1敗グループ】【1勝2敗グループ】…

STEP 3: ペアリング生成
  - スコアの高いグループから順にマッチングを試みる
  - 同一スコア内で過去に対戦していないプレイヤー同士をペアにする
  - ペアリングの選択方式（実装は以下のいずれかを選択）:
      a) ランダム方式: 同スコア内でシャッフルして順にペアを作成
      b) ダッチシステム: スコア上位同士を優先して組み合わせる
      c) 重み付きグラフの最大重みマッチング（大規模大会向け）

STEP 4: 調整処理（ペアが組めない場合）
  - 奇数人数や再戦バッティングでペアが組めない場合:
      → 隣接するスコアグループからプレイヤーを繰り上げて調整する
  - 大規模大会・組み合わせ爆発が懸念される場合:
      → モンテカルロ法を応用し、ランダムに複数の対戦表を生成
      → 全制約（再戦回避・BYE制限）を満たすスコアが最高のものを採択する

STEP 5: BYE 処理
  - 奇数参加者が残った場合、最下位スコアのプレイヤーに BYE を割り当てる
  - ただし、すでに BYE を受けたプレイヤーは除外する
  - BYE を受けたプレイヤーは勝利扱いとして勝ち点を付与する

STEP 6: 卓番号割り当てと DB 登録
  - ペアが確定したら卓番号を割り当てる（1卓から昇順）
  - match_game テーブルにレコードを挿入する
```

#### 5.1.3 実装上の注意

- 再戦チェックは `match_game` テーブルを参照し、同大会内の全ラウンドを対象とする
- 理論上、再戦を完全に回避できないケースでは管理者に警告を表示し、手動調整を促す
- BYE の割り当て対象は「BYE 未経験者の中で最も勝ち点が低いプレイヤー」とする

### 5.2 ランキング計算ロジック

ポケカ公式スイスドローのランキング決定手順に準拠する。

#### 5.2.1 勝ち点（Match Points）

```
計算方法:
  勝利   : +3 点
  負け   :  0 点
  引き分け:  0 点（時間切れ引き分けも同様。両者 0 点扱い）

全ラウンド終了時の累計勝ち点で一次順位を決定する（降順）。
```

#### 5.2.2 OMW%（Opponent Match Win %）：オポネント・マッチ・ウィン・パーセンテージ

```
概要:
  「大会を通じて、自分の対戦相手がどれだけ勝ったか」の平均割合。
  数値が高いほど強い相手と戦ったことを示し、上位判定に有利に働く。

計算手順:
  1. 自分の各対戦相手について個別の勝率（MWP）を計算する:
       MWP_相手 = 相手の獲得勝ち点 / 相手の最大獲得可能勝ち点
               = 相手の勝ち点 / (相手の試合数 × 3)

  2. 底上げ（フロア）ルールを適用する:
       MWP_補正 = MAX(MWP_相手, 0.33)   ← 33% を下限として切り上げ
       ※ BYEによる試合は相手の勝ち点計算から除外する

  3. OMW% = 全対戦相手の MWP_補正 の平均
       OMW% = Σ(MWP_補正) / 対戦した相手の人数

勝ち点が同点の場合、OMW% の高い方が上位となる。
```

#### 5.2.3 タイブレイカー（OMW% も同点の場合）

以下の指標を順番に比較する:

| 優先順位 | 指標名 | 内容 |
|---|---|---|
| 1位 | 勝ち点 | 累計勝ち点（3点/勝利） |
| 2位 | OMW%（オポ） | 対戦相手の勝率平均（下限33%） |
| 3位 | 対戦相手の獲得勝ち点合計 | 自分の全対戦相手の勝ち点を単純合計 |
| 4位 | OOMW%（オポのオポ） | 対戦相手が対戦した相手の強さ（OMW%）の平均 |
| 5位 | 直接対決の結果 | 同点者同士が直接対戦している場合のみ適用 |

```
※ 4位・5位まですべて同率の場合は同順位とする。
※ OMW% / OOMW% の計算はアプリケーション側（Java の RankingService）で行う。
※ BYE の勝ち点（3点）は自分の勝ち点には加算するが、
   相手（BYE）の MWP 計算には含めない（対戦相手扱いしない）。
```

#### 5.2.4 勝ち点設定について

大会作成時に管理者が以下を設定できる（デフォルト値は上記ポケカ標準）:

| 設定項目 | デフォルト | 説明 |
|---|---|---|
| 勝利ポイント | 3 | 勝利時に付与する勝ち点 |
| 敗北ポイント | 0 | 敗北時に付与する勝ち点 |

### 5.3 決勝トーナメント生成ロジック

```
設定：
  - 参加者総数（例：32人）
  - 1トーナメントの人数（例：8人）

フロー：
  1. スイスドローランキング順に参加者をソート
  2. 設定人数ごとにグループ分け
     例: 32人 / 8人 = 4グループ (1-8, 9-16, 17-24, 25-32)
  3. 各グループ内で単純トーナメント表を生成
  4. 試合を管理（勝者を次ラウンドへ進める）
```

### 5.4 QRコードによるチェックイン

```
フロー：
  1. 参加者はエントリー後、自身の画面でQRコードを表示
     QRコードには「大会ID + プレイヤーID + ハッシュ（改ざん防止）」を埋め込む
  2. 運営者がQRコードリーダー画面でスキャン
  3. サーバーにてデコード・検証
  4. 該当プレイヤーのチェックインステータスを更新
```

### 5.5 ポイント計算ロジック

```
大会終了時：
  1. 決勝トーナメント結果から最終順位を決定
  2. 設定された入賞ポイント（1位～16位）に対応するポイントを付与
  3. 全参加者に参加ポイントを付与
  4. player_point テーブルにポイントを記録（累計更新）
  5. エントリーに account_id が紐づいている場合、アカウント別累計ポイントも更新
```

### 5.6 アカウント機能ロジック

```
【初回アクセス時】
  1. ブラウザに Cookie が存在しない場合 → アカウント登録/ログイン画面を案内（強制ではなくオプション）
  2. ユーザーが「アカウント登録」を選択 → 名前・パスワードを入力して登録
  3. 登録完了後、自動ログイントークン（UUID）を生成し Cookie にセット（有効期限 120日）
  4. DBの player_account_token テーブルにトークンを保存

【次回以降のアクセス】
  1. Cookie の自動ログイントークンを確認
  2. DBと照合してトークンが有効 かつ 期限内であれば自動ログイン
  3. 期限切れの場合はトークンを削除して未ログイン状態に戻す

【エントリー（ログイン済み）】
  即エントリー:
    → アカウントの登録名（player_account.display_name）でエントリー登録
    → 確認画面なし・ワンクリック完結
  エントリー名を指定してエントリー:
    → エントリー名入力画面に遷移
    → 任意の名前でエントリー（アカウント名と異なっていても可）
    → entry.account_id にログイン中のアカウントIDを紐づけて保存

【エントリー（ゲスト）】
  → 従来通り名前を入力してエントリー（entry.account_id = NULL）

【アカウントと大会名の関係】
  - player_account.display_name: アカウント表示名（変更可能）
  - entry.player_name: 各大会のエントリー名（大会ごとに異なっていい）
  - 同一人物の識別には account_id を使用する
  - 過去大会履歴・累計ポイントは account_id ベースで集計する
```

### 6.2 参加者アカウント認証

- **パスワードハッシュ**: BCrypt でハッシュ化して保存
- **自動ログイン**: UUID v4 のランダムトークンを Cookie にセット（`HttpOnly`, `SameSite=Lax`）
  - Cookie 名: `APT_TOKEN`
  - 有効期限: 120日（DBのexpire_atと照合）
  - トークンはログアウト・期限切れで無効化
- **アカウントとエントリーの分離**: `entry.player_name` と `player_account.display_name` は独立管理  
  ユーザーの同一性は `entry.account_id` （FK to `player_account.id`）で管理

### 6.3 管理者認証

- Spring Security によるフォームログイン認証
- 管理者アカウントは DB 管理（`admin_user` テーブル）
- セッション管理は Tomcat のデフォルト設定
- `/admin/**` 配下は管理者ロールのみアクセス可能

### 6.4 QRコード改ざん防止

- QRコードのペイロードに HMAC-SHA256 ハッシュを付加
- サーバー側でハッシュ検証を実施
